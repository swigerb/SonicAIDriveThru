## Learnings

<!-- Append new learnings below. Each entry is something lasting about the project. -->
- **Asyncio shared state safety**: mutable state shared between coroutines (like `ai_speaking` and `cooldown_end`) is safe in asyncio without locks — event loop is single-threaded, all state checks atomic within tick.
- **gpt-realtime-1.5 prompt best practices**: Bulleted format with ALL CAPS emphasis beats prose paragraphs for instruction following. Explicit phrase variety rules prevent bot-like repetition. Reduce prompt token count through conciseness.
- **WebSocket message ordering**: Client messages arrive ordered via WebSocket. Server must forward session.update (with tools + system message) before greeting, or OpenAI processes greeting without context.
- **Menu category mapping coupling**: `_infer_category()` in tools.py couples to frontend menuItems.json. ALLOWED/BLOCKED lists must include both canonical category names AND keyword-inferred fallbacks.
- **Python notebook programmatic editing**: `.ipynb` files have complex JSON escaping — use Python JSON manipulation instead of text-based edit tools.
- **Search index correctness**: If `.env` points to wrong index, no search results. Always verify AZURE_SEARCH_INDEX env var matches expected index name.
- **System prompt token accounting**: Closing phrases consume ~20 tokens (e.g., "Your carhop will have that right out to you!"). `max_tokens=150` too aggressive for order recap + closing. 250 is demo-safe.
- **Order quantity limits**: Added `MAX_QUANTITY_PER_ITEM = 10` and `MAX_TOTAL_ITEMS = 25` in `tools.py` to prevent abuse. Validation runs in `update_order` before `handle_order_update`, only on "add" actions. Per-item check uses item+size combo match. Error messages are warm/drive-thru-friendly and sent `TO_SERVER` so the AI relays them naturally. Constants at module top for easy tuning. Defense-in-depth with Unity's conversational guardrails.
- **ToolResultDirection routing for tool outcomes**: Successful `update_order` must use `TO_BOTH` (not `TO_CLIENT`). `TO_CLIENT` sends empty string to AI model — it gets no confirmation order succeeded and goes silent. `TO_BOTH` sends JSON to both OpenAI and frontend. Error/limit responses use `TO_SERVER` (AI relays message). This distinction is critical for multi-item order flow.
- **Hardened price validation**: Zero-dollar items rejected early in `update_order` with a friendly retry message (`TO_SERVER`). Catches model hallucination when it skips search and guesses $0.00. Must be first guard in the add path — before extras and quantity checks.
- **Combo detection pending slots**: When a Combo item is added, a `(COMBO DETECTED: ...)` hint is appended to the ToolResult telling the AI it MUST ask for side and drink selections. Case-insensitive substring match on "combo" in item_name.
- **Human-readable size formatting**: Search results now parse the `sizes` JSON field into "Small ($X.XX), Medium ($Y.YY)" format. Falls back to raw string if parsing fails. Helps gpt-realtime-1.5 speak prices naturally instead of reading raw JSON.
- **Upsell hints in tool results**: After successful `update_order`, category-based hints appended to ToolResult (e.g., combo upgrade, combo conversion for burgers, flavor add-in for drinks). Uses `_infer_category()` — must keep category lists in sync with both canonical and keyword-inferred names.
- **Combo validation via `get_combo_requirements()`**: Replaces ad-hoc combo detection with a persistent order-wide scan. Counts combos vs sides vs drinks across the whole order using `_infer_combo_component()` in order_state.py (avoids circular import with tools.py). Returns `is_complete`, `missing_items`, and `prompt_hint`. The `[SYSTEM HINT: ...]` pattern in tool results tells the AI exactly what to ask next. Upsell hints are suppressed while combo is incomplete — focus on completion first.
- **Delta summaries for voice vs screen**: `ToolResult` now supports separate `client_text` for the frontend. Server gets a natural-language delta ("Added 1 Large Cherry Limeade — your total is now $8.49") while the client gets pure JSON for the order display panel. `to_client_text()` falls back to `to_text()` for backward compatibility.
- **Sonic-specific size formatting**: Removed Dunkin' remnants (Kannchen, Pot) from `handle_order_update`. Added Route 44 support (`rt44`, `rt 44`, `route 44`). Unrecognized sizes now default to empty string (hidden in display) rather than capitalized — prevents random size strings from polluting the order display.
- **Greeting-before-session.update caused tool blindness**: `from_client_to_server()` sent `response.create` (greeting) BEFORE forwarding the client's `session.update` (which carries tools). OpenAI generated the greeting without tools configured, and depending on model behavior, tools could remain unregistered for subsequent turns. Fix: forward the client message first, then send the greeting. This was documented as a learning ("Server must forward session.update before greeting") but never actually applied in code.
- **Defense-in-depth tools_pending registration**: Added fallback registration of `call_id` in `response.output_item.added` (not just `conversation.item.created`). Prevents silent tool call drops if event ordering changes across API versions. `conversation.item.created` always overwrites with the correct `previous_item_id`.
- **Order panel fix via message reordering**: The fix to "greeting before session.update" was a simple message-ordering bug in the WebSocket relay. This is the correct final fix. Previous learning about needing to forward session.update before greeting was correct principle, just not implemented until now.
- **Greeting prompt must be imperative, not descriptive**: "Greet the guest: '...'" causes gpt-realtime to generate meta-commentary about HOW to greet. Must use direct imperative: "Greet them now. Say ONLY the greeting itself." The model treats quoted examples as illustrative rather than prescriptive.
- **Pre-set ai_speaking before response.create**: There's a timing gap between sending `response.create` and receiving the first `response.audio.delta` where mic audio can leak through echo suppression. Pre-setting `ai_speaking = True` in `send_greeting_once()` closes this gap. The flag is naturally reset by `response.audio.done` later.
- **Echo cooldown 300ms too short for startup**: The initial greeting is the worst-case echo scenario — speakers at full volume, no prior calibration. 500ms cooldown provides sufficient margin for speaker echo to decay without noticeable delay to the user.
- **Dual-trigger greeting for API resilience**: Azure OpenAI Realtime API may not always send `session.updated` after `session.update`. Relying solely on server-side `session.updated` event caused complete silence on startup. Fix: dual-trigger — primary fires on `session.updated` in `from_server_to_client` (preferred, confirms tools configured), fallback fires after forwarding `session.update` in `from_client_to_server`. The existing `greeting_sent` flag prevents double-greeting. Always design for "whichever fires first wins" when depending on external API events.
