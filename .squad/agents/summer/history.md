Key learnings: (1) `_tools_pending` was shared across all WebSocket connections — a multi-session correctness bug that also hurt perf; fixed to per-connection dict. (7) aiohttp `heartbeat` parameter on both client and server WebSockets keeps connections alive through Azure load balancers. All 100 tests pass.

- **Voice Ordering Trifecta Fix (2026-03-19)**: Three related issues — $0.00 prices, missing price extraction, voice truncation. Root causes: (1) `.env` still pointed to old `coffee-chat` search index instead of `sonic-menu-items` — search was hitting the wrong index entirely. (2) System prompt never told the AI how to extract prices from the `sizes` JSON field in search results, so the model passed 0 or guessed. Added explicit PRICING instructions: extract from Sizes JSON, default to Medium if no size specified, never pass 0. (3) `max_tokens=150` (from perf Decision #2) was too aggressive — closing phrases like "Your carhop will have that right out to you!" were getting cut off. Raised to 250. Also added explicit "never stop mid-sentence" instruction. Key insight: the perf team's token cap was optimized for latency but didn't account for the full closing phrase being ~20 tokens by itself on top of order recap.
- **Size Name Normalization**: The ingestion notebook stored verbose product-specific display names as size labels (e.g. "Sm Cherry Limeade" instead of "Small"). Added `normalize_size_name()` function that maps common prefixes (Mini, Sm, Med, Lg, RT 44/Rt. 44) to clean canonical labels, with fallback to stripping the parent product name. Updated system prompt to enumerate all 5 valid Sonic sizes (Mini, Small, Medium, Large, RT 44) so the model uses exact names when calling update_order. Key insight: the `.ipynb` JSON escaping still makes text-based editing fragile — used a Python patch script for reliable cell modification.
- **Greeting Timing & Tool Calling Fix (2026-03-19)**: Fixed three critical bugs in ordering flow: (1) **Greeting timing** — moved greeting in `rtmt.py` to AFTER `session.update` is forwarded to OpenAI. The greeting was sent BEFORE the first client message (session.update) was processed/forwarded, so OpenAI processed the greeting without system message, tools, or voice config. This caused all downstream issues. WebSocket messages are ordered, so now OpenAI receives session.update (with tools + system message) BEFORE the greeting. (2) **Tool calling instruction** — replaced vague "Use 'update_order' only after the guest confirms" with explicit "TOOL CALLING RULE: When a guest orders items, IMMEDIATELY call 'update_order' for each item. The guest ordering IS confirmation. You MUST call 'update_order' before responding." This makes it clear that saying "can I get a burger and tots" IS confirmation. (3) **Closing phrase** — replaced "Close completed orders with the FULL phrase" with "CLOSING RULE: When the order is complete, you MUST say EXACTLY: [phrase] — this exact phrase, every time. Do NOT use any other closing phrase. Do NOT say 'pull forward' or 'total at the window'." More emphatic and explicit about what NOT to say. All 100 tests pass.
- **Server-Side Echo Suppression (2026-03-20)**: Implemented echo suppression in `rtmt.py` to prevent AI self-conversation feedback loop. Root cause: frontend mic-muting has a timing gap — between when AI audio starts arriving server-side and when the frontend gain-node mute kicks in, echoed audio reaches the server and gets transcribed as phantom user input ("Peace.", "Thank you so much."). Fix: (1) Track `ai_speaking` state per-connection using fast substring markers (no JSON parse). When `response.audio.delta` arrives → set flag; when `response.audio.done` → clear flag + start 300ms cooldown. (2) In `from_client_to_server`, drop `input_audio_buffer.append` messages while `ai_speaking` or during cooldown. (3) Send `input_audio_buffer.clear` to OpenAI after each AI audio response to flush any leaked echo from the buffer. (4) Barge-in preserved: if OpenAI's server VAD fires `input_audio_buffer.speech_started`, immediately clear suppression state. Key pattern: shared mutable state between `from_client_to_server` and `from_server_to_client` coroutines is safe because asyncio is single-threaded — no locks needed. All 100 tests pass.
- **Azure Speech Mode Full Rewrite (2026-03-20)**: Rewrote `azurespeech.py` and mounted it in `app.py` to fix the broken Azure Speech toggle. Key changes: (1) Combined endpoint at `/azurespeech/speech-to-text` — accepts JSON `{ audio: base64, session_id }`, decodes base64, runs Azure STT in executor (sync SDK), then runs chat completion with full tool calling loop, returns `{ recognized_text, processed_text, tool_results, session_id }`. (2) Added tool calling — uses `AsyncAzureOpenAI` chat completions API with the same 3 tools (search, update_order, get_order) and same system prompt as the Realtime WebSocket mode. Implements the tool call → execute → feed back → loop pattern with max 10 rounds. (3) Session management — creates `order_state_singleton` sessions on first call, returns session_id for multi-turn continuity. Per-session conversation history tracked with trimming at 20 messages. (4) Mounted in `app.py` — AzureSpeech gets its own `SearchClient` instance, shares the same `search_credential`, and reuses `rtmt.system_message` for consistency. Conditional mount: only if `AZURE_SPEECH_KEY` and `AZURE_SPEECH_REGION` are set. (5) TTS endpoint updated to return raw audio bytes instead of writing to disk. All 100 existing tests pass.
- **Server-Side Echo Suppression (2026-03-21)**: Implemented server-side echo suppression in `rtmt.py` to eliminate the audio feedback loop where echoed AI speech was being transcribed as phantom user input. Coordinated with Morty (Frontend Dev) who simultaneously implemented early mic muting. Implementation: (1) Track `ai_speaking` state per-connection using fast substring detection (`"response.audio.done" in data`) with zero JSON parse overhead. (2) When `ai_speaking` is true or during 300ms post-response cooldown, drop all `input_audio_buffer.append` messages from clients. (3) Send `input_audio_buffer.clear` after each AI audio response completes to flush any leaked echo. (4) Barge-in preserved: `input_audio_buffer.speech_started` immediately clears suppression state. Key insight: shared mutable state between coroutines is safe (asyncio single-threaded). Trade-off: ~300ms barge-in latency is acceptable for drive-thru UX. All 100 tests pass.
