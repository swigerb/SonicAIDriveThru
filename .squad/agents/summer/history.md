- **Sonic Rebrand (backend)**: Replaced all Dunkin references across backend Python files, system prompts, tools, test suites, docs, and menu data. Key insight: the `_infer_category()` function in tools.py is tightly coupled to the frontend menuItems.json — the `MENU_CATEGORY_MAP` loads at module init from that file, so ALLOWED_EXTRA_CATEGORIES and BLOCKED_EXTRA_CATEGORIES must include both the frontend JSON category names AND the fallback keyword-inferred names. The structured_menu_items file was malformed JSON with Dunkin data; replaced entirely with Sonic menu items (slushes, shakes, burgers, hot dogs, tots, breakfast, drinks, extras).

- **Team Orchestration (2026-03-19T04-06)**: Rick provided scope analysis, Morty completed frontend rebrand, Summer completed backend rebrand (69 tests pass), Birdperson created verification tests (12 tests pass).
- **Production Performance Hardening**: Major perf pass across all backend files. Key learnings: (1) `_tools_pending` was shared across all WebSocket connections — a multi-session correctness bug that also hurt perf; fixed to per-connection dict. (2) Audio delta messages (`response.audio.delta`, `response.audio_transcript.delta`) are the hottest path in rtmt.py — added `_PASSTHROUGH_TYPES` frozenset for O(1) early-return before match/case. (3) `__slots__` on ToolResult/Tool/RTToolCall reduces per-object memory. (4) `FileResponse` doesn't have `.body` attribute — compression middleware must guard with `isinstance(response, web.Response)`. (5) Module-level search cache (`_SearchCache`) needs `clear()` for test isolation since tests share the same process. (6) `order_state.py` logging at INFO was serializing full OrderSummary on every `get_order_summary` call — demoted to DEBUG. (7) aiohttp `heartbeat` parameter on both client and server WebSockets keeps connections alive through Azure load balancers. All 100 tests pass.

- **Sonic Menu Ingestion Notebook**: Created `scripts/sonic_menu_ingestion_search.ipynb` based on the existing `menu_ingestion_search_json.ipynb`. Key differences: hardcodes `sonic-menu-items` as the index name (not read from env), reads from `app/frontend/src/data/sonicmenuitems.json`, all markdown/comments reference Sonic Drive-In, search tests use Sonic-relevant queries (burgers, slushes, shakes). Also updated `.env-sample` and `infra/main.parameters.json` default index names from `sonic-drive-in`/`voicerag-intvect` to `sonic-menu-items`. The `app.py` reads the index name from `AZURE_SEARCH_INDEX` env var so no code change needed there — just ensure the .env has the correct value.
- **Production Menu Ingestion Refactor**: Refactored `scripts/sonic_menu_ingestion_search.ipynb` to parse the production `sonic-menu-items.json` (nested menus/products/categories/productGroups format). Key patterns: (1) Recursive category traversal — categories can nest (`categories.` childRefs point to subcategories), must recurse to find leaf products. (2) Only 172 of 1334 products are referenced from categories; all 172 are non-recipe. (3) Size variants resolved via `relatedProducts.alternatives` → `productGroups` → `childRefs` → individual product prices. 86 have size alts, 86 don't (get Standard size). (4) Top-level categories identified by excluding any category referenced as a child. Only 1 top-level in national menu. (5) Added `image` field to index schema. (6) Batched embedding generation (batch_size=100) as safety for larger menus. (7) `.ipynb` files are best edited programmatically via Python JSON manipulation rather than text-based edit tools — the JSON string escaping makes text matching unreliable. (8) Legacy coffee-specific fields (origin, caffeineContent, brewingMethod, popularity) kept in schema for backward compat but set to empty string.
- **Voice Ordering Trifecta Fix (2026-03-19)**: Three related issues — $0.00 prices, missing price extraction, voice truncation. Root causes: (1) `.env` still pointed to old `coffee-chat` search index instead of `sonic-menu-items` — search was hitting the wrong index entirely. (2) System prompt never told the AI how to extract prices from the `sizes` JSON field in search results, so the model passed 0 or guessed. Added explicit PRICING instructions: extract from Sizes JSON, default to Medium if no size specified, never pass 0. (3) `max_tokens=150` (from perf Decision #2) was too aggressive — closing phrases like "Your carhop will have that right out to you!" were getting cut off. Raised to 250. Also added explicit "never stop mid-sentence" instruction. Key insight: the perf team's token cap was optimized for latency but didn't account for the full closing phrase being ~20 tokens by itself on top of order recap.
- **Size Name Normalization**: The ingestion notebook stored verbose product-specific display names as size labels (e.g. "Sm Cherry Limeade" instead of "Small"). Added `normalize_size_name()` function that maps common prefixes (Mini, Sm, Med, Lg, RT 44/Rt. 44) to clean canonical labels, with fallback to stripping the parent product name. Updated system prompt to enumerate all 5 valid Sonic sizes (Mini, Small, Medium, Large, RT 44) so the model uses exact names when calling update_order. Key insight: the `.ipynb` JSON escaping still makes text-based editing fragile — used a Python patch script for reliable cell modification.
